# 📱 호남대학교 축제 실시간 POS & 모바일 오더기 시스템

본 시스템은 **대학 축제 주점 환경(1대의 마스터 POS 단말기 + 7대의 서빙 직원 스마트폰 오더기)**에서 동시에 많은 트래픽과 데이터 변경이 발생하더라도 락 에러나 데이터 유실 없이 실시간으로 작동하도록 설계된 고동시성 지원 실시간 POS 솔루션입니다.

## 🗄️ 2. 데이터베이스(DB) 테이블 스키마 구조

SQLite3 데이터베이스(`pos.db`)는 총 9개의 테이블로 유기적으로 연결되어 동작합니다.

| 테이블명 | 주요 컬럼 구성 | 역할 및 특징 |
|---|---|---|
| **`tables`** | `table_id`, `status`, `price`, `paid_price`, `menus`, `paid_menus` | 테이블별 현재 이용 상태, 조리 대기 품목/금액 및 서빙 완료 품목/정산 금액을 분리 보관 |
| **`pending_orders`** | `order_id`, `table_id`, `timestamp`, `items`, `checked_items` | 주방 POS 화면에 대기 순으로 뿌려지는 개별 주문서(티켓). 완료/취소 시 소거됨 |
| **`sales_summary`** | `id` (항상 1), `total_sales`, `item_sales` | 오늘 하루 동안 서빙 완료된 실시간 누적 총 매출액 및 품목별 누적 판매량 보관 |
| **`daily_archive`** | `id`, `date`, `sales`, `details` | 일차 마감 시 해당 날짜의 총 매출액과 상세 판매량 장부를 영구 박제(아카이브) |
| **`order_history`** | `id`, `table_id`, `time`, `items`, `cost` | 주문 추가, 품목 완료, 취소 등 모든 변동 이력을 로그로 기록하는 역추적 장부 |
| **`away_timers`** | `table_id`, `time_left`, `started_at`, `running` | 자리비움 타이머 상태. 절대 서버 스탬프(`started_at`)를 기록해 새로고침 시에도 정밀 유지 |
| **`sold_out`** | `menu_name`, `is_sold_out` | 메뉴 품절 여부 보관. 품절 시 직원 오더기에서 선택 불가 상태로 자동 동기화 |
| **`waiting_list`** | `id`, `name`, `phone`, `people`, `status`, `timestamp` | 현장 대기자 명단. 등록(waiting) ➡️ 호출(called) ➡️ 착석(seated) / 취소(cancelled) 흐름 제어 |
| **`checked_items`** | `table_id`, `items_json` | (하위 호환성 유지용 임시 테이블) |

## 🛡️ 3. 고부하 트래픽 및 동시성 락 방어 설계

7명의 직원이 동시에 주문을 누르고 대기를 입력할 때 발생하는 SQLite 파일 락 에러(`database is locked`) 및 데이터 유실(Lost Update) 문제를 원천 예방하기 위해 아래 기술들이 적용되어 있습니다.

1. **고속 동시성 WAL(Write-Ahead Logging) 모드 강제**
   - 데이터베이스 접속 시 상시 WAL 저널 모드를 적용하여, 쓰기 작업(주문/대기 등록)이 수행되는 도중에도 읽기 작업(주방 POS 화면 갱신)이 차단(Lock)되지 않고 서로 독립적으로 고속 처리됩니다.
2. **락 대기 타용아웃 확장 (5초 ➡️ 30초)**
   - 동시 트랜잭션 폭주로 데이터베이스에 일시적인 대기열이 생기더라도, 에러를 반환하지 않고 최대 30초간 대기하며 순차적으로 디스크에 기록합니다.
3. **IMMEDIATE 트랜잭션 (`BEGIN IMMEDIATE`) 의무화**
   - 주문 접수, 서빙 완료, 대기 접수 등 쓰기(CUD) 작업을 수반하는 16개 모든 소켓 이벤트 핸들러 시작부에 즉시 쓰기 락(Reserved Lock)을 획득하도록 구현하여, 동일 테이블 조회 후 덮어써져 주문이 유실되는 동시성 버그를 원천 차단합니다.

## 📋 4. 축제 현장 실전 운영 지침 및 주의사항 (필독)

하드웨어와 네트워크 한계를 현장 인력 가이드라인으로 무력화하는 핵심 수칙입니다.

* **서빙 직원 역할 제한 (주문 충돌 원천 방지)**
  - 홀 서빙 직원들은 스마트폰 오더기로 **"오직 신규 주문 추가"**만 처리하도록 역할을 국한합니다.
  - 이미 조리 중이거나 서빙 완료된 품목을 감산(-)하거나 취소하는 등의 예외 조작은 주방/카운터의 **마스터 POS 담당자(1명)에게 무전이나 구두로 요청하여 처리**하게 교육합니다.
* **대기 관리자 단일화**
  - 대기자 명단을 호출하고 착석 유도를 진행하는 직원은 **입구에 배치된 단 1명의 직원**으로 지정하여, 착석 처리와 취소 처리가 중복 엇갈리는 현상을 예방합니다.
* **오더기 스마트폰 설정 가이드**
  - **화면 자동 꺼짐 해제**: 스마트폰 화면이 꺼지면 브라우저 소켓 통신이 끊겨 재연결 지연이 생깁니다. 모든 직원 폰의 화면 자동 꺼짐 시간을 **"최대(30분 이상)"**로 설정하도록 지시하십시오.
  - **보조 배터리 필수 지참**: 화면이 계속 켜진 채 실시간 통신을 하므로 배터리 소모가 빠릅니다. 직원당 보조 배터리 1개씩을 필수 지참하게 하십시오.
* **독립 와이파이(AP) 구축**
  - 외부 축제객들이 무단 연결하여 IP 대여 풀이 고갈되는 것을 막기 위해, 매장용 공유기의 **SSID를 숨김(비공개) 처리**하고 직원 기기들만 사설망 비밀번호를 등록해 사용하십시오.

## ⚡ 5. 네트워크 접속 및 무설치 앱(PWA) 설치법

### 💡 접속 속도 가이드
- **추천 (내부 IP 접속)**: 노트북(서버)과 폰들을 동일 공유기에 묶고, 노트북의 로컬 IP(예: `http://192.168.0.25:3000`)로 직접 접속하십시오. 외부 망을 타지 않아 렉이 전혀 없이 **0.05초 미만의 실시간 속도**를 자랑합니다.
- **차선 (ngrok 외부 주소)**: 외부 통신망 사정상 로컬 IP 접속이 어려울 경우 ngrok 도메인(`https://xxxx.ngrok-free.dev`)으로 접속합니다. 셀룰러 혼잡도에 따라 약 **0.5초 ~ 1.5초**의 로딩 딜레이가 발생할 수 있습니다.

### 📱 5초 만에 주소창 없는 전체화면 앱(PWA) 만들기
직원 스마트폰 및 주방 태블릿에 거추장스러운 웹 브라우저 주소창을 없애고 완벽한 네이티브 앱처럼 설치하는 방법입니다. (APK 빌드 필요 없음)

1. 안드로이드/iOS 기기에서 **Chrome(크롬) 또는 Safari(사파리) 브라우저**를 엽니다.
2. 마스터 POS의 경우 `접속주소/pos.html`, 오더기의 경우 `접속주소/`를 입력해 접속합니다.
3. 브라우저 메뉴(점 3개 또는 공유하기 버튼)를 누릅니다.
4. **`[홈 화면에 추가]`** 또는 **`[앱 설치]`** 버튼을 터치합니다.
5. 바탕화면에 바로가기 앱 아이콘이 생성되며, 이를 터치하면 주소창이 없는 **100% 가득 찬 네이티브 앱 형태로 즉시 가동**됩니다.
   - *이 앱 모드로 구동할 때 주방 신규 주문 알림음(.mp3) 오토플레이가 가장 매끄럽게 작동합니다.*
